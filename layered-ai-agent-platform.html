01 — The Problem

Why Most LLM Systems Break in Production

There's a gap between an LLM demo that works and an LLM system that's safe, reliable, and maintainable in production. The gap is almost entirely architectural.

The failure pattern is consistent: a single file with an API call, some tools bolted on, a system prompt that says "only respond about X", and the belief that the model will behave because you asked it to.

The core problem

Prompt instructions are suggestions, not enforcement. A Pydantic validator that rejects a malformed output cannot be jailbroken. A tool that runs Python cannot hallucinate numbers. These are fundamentally different guarantees — and most LLM projects treat them as equivalent.

The Layered AI Agent Platform is built around one idea: every concern lives in its own layer with an explicit contract about what it does and what it never does.

02 — System Architecture

The name "Layered" isn't marketing — it's the entire design philosophy. No leakage between layers. No shortcuts. Every layer is independently testable and replaceable.

04 — Request Lifecycle

What Happens on Every Query

Example: "How does my usage this month compare to last month?"

01
Request received
POST /api/v1/query. Middleware attaches trace_id. Pydantic validates schema.
api/
02
Input guardrails
PII scan. Policy check for blocked intents. Blocks before any orchestration begins.
guardrails/input
03
Query planning
Planner classifies: MIXED. Selects tools: [data_summary, trend_comparison]. Flags memory needed.
orchestration/planner
04
Tool execution
data_summary(current_month) → exact totals. trend_comparison(current, prior) → delta + direction. Pure functions.
tools/
05
Memory context loaded
User profile retrieved. Session history: last 10 turns. Context assembled deliberately, not blindly injected.
memory/
06
Context assembly → LLM call
PromptBuilder assembles: system_prompt + tool_results + memory + query. LLM receives grounded facts. It writes — it does not compute.
llm/
07
Output guardrails
Schema check: required fields present. Number grounding: values in response match tool output.
guardrails/output
08
Observability + response
Full trace emitted: trace_id, tool calls, token usage, latency_ms. Response: {answer, structured_data, citations}.
observability/
Key insight

The LLM is step 6 out of 8 — not step 1. The orchestrator assembles everything before the LLM sees a single token. It receives fully grounded, fact-loaded context and writes language around it. It never retrieves, computes, or routes.

05 — Design Principles

The Decisions That Actually Matter

01
LLM is a reasoning layer
It writes language. It never computes, retrieves, or plans. Every number traces to a deterministic tool call.
02
Tools own the numbers
Tools are pure functions. Same input, same output always. Unit tested with exact assertions — zero tolerance.
03
RAG over fine-tuning
Knowledge updates without retraining. Every claim has a citable source. Fine-tuning bakes knowledge in with no audit trail.
04
Guardrails are code
Pydantic validators enforce constraints regardless of what the LLM produces. Prompt instructions are suggestions. Code is enforcement.
05
Memory is not one thing
Session context ≠ user profile ≠ summarized history. Different lifecycles, different stores. Conflating them creates stale bugs.
06
Prompts versioned like artifacts
Prompts live in versioned directories. Promotion requires eval suite pass. No prompt ships without data behind it.
07
Evaluation from day one
4-tier scoring built into CI/CD. Hallucination rate is a tracked health metric — not an afterthought.
08
Observability from day one
Every step emits a structured JSON event with trace_id, duration, and context. OTel-ready. You always know what happened.
09
Architecture over feature bloat
The value is in interfaces, layering, and decision-making — not line count. Every choice has a documented rationale.
06 — Why the LLM Never Does Arithmetic

Tools Layer Deep Dive

LLMs are probabilistic text predictors — not reliable calculators. A wrong number looks identical to a right one. No error, no exception, no warning. Every tool is a pure Python function: deterministic, side-effect-free, unit-testable with exact assertions.

def data_summary(
    dataset: DataSet,
    period: Optional[str] = None,
) -> ToolResult:
    """Pure function — same input always returns same output."""
    records = dataset.records
    if period:
        year, month = map(int, period.split("-"))
        records = [r for r in records
                   if r.date.year == year and r.date.month == month]

    total = sum((r.amount for r in records), Decimal("0"))
    return ToolResult(
        tool_name="data_summary",
        success=True,
        data={"total": str(_round(total)), "count": len(records)},
    )
def test_data_summary_monthly():
    result = data_summary(sample_dataset, period="2024-01")
    assert result.data["total"] == "1305.89"  # exact — zero tolerance
07 — Enforcement at the Boundary

Guardrails as Code, Not Prompts

Why not just add this to the system prompt?

"Always respond in JSON" in a system prompt is a request. A Pydantic model that raises ValidationError when required fields are missing is a guarantee. One can be bypassed by adversarial input. The other cannot.

Guardrails run twice — on input before orchestration begins, and on output before the response reaches the user. Input covers PII detection (SSN, card numbers, phone via regex), policy rules, and schema validation. Output covers required fields, response length, and number cross-referencing against tool outputs.

09 — Architecture Decision Records

Every Major Decision Documented

ADR-001Accepted
Use RAG over fine-tuning for domain knowledge
Domain knowledge changes frequently. RAG allows document updates without model retraining. Citations are first-class — every fact has a traceable source. Fine-tuning costs 10–100x more and produces knowledge that can't be audited or updated without a full retrain.
ADR-002Accepted
Deterministic tools for all data computations
LLMs are unreliable calculators. Tools are pure functions: deterministic, unit-testable, auditable. All computed values trace to tool outputs, enabling hallucination detection at the output guardrail layer.
ADR-003Accepted
Separate short-term and long-term memory
Session context is ephemeral and cheap — fast in-memory access. User profile is durable — Redis. Different lifecycles, different storage requirements. Conflating them creates stale context bugs and wasted tokens per request.
ADR-004Accepted
Guardrails as a separate architectural layer
Prompt-based safety is a suggestion, not enforcement. Code-level guardrails cannot be bypassed by user input. Separate layer means independent iteration, testing, and a clear audit trail — every blocked request logged with the specific rule triggered.
ADR-005Accepted
Evaluation as a first-class engineering concern
Teams that defer evaluation discover regressions in production. Four-tier framework from day one: deterministic CI gate, retrieval quality, LLM response quality, prompt regression. Hallucination rate is a tracked metric with an explicit target: below 2%.
